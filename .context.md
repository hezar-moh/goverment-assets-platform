## Key Architectural Decisions

1. **Multi-tenancy via django-tenants** — Each ministry gets its own PostgreSQL schema (`moh_schema`, `mof_schema`, etc.). Schema isolation at database level.
2. **JWT + session dual auth** — Mobile app uses JWT tokens (API). Web browser uses Keycloak SSO + Django sessions.
3. **Keycloak for SSO** — Keycloak handles passwords. Django never stores passwords. Flows: web SSO, API login via custom backend, API token-based (mobile).
4. **Role-Based Access Control** — 5 roles: SUPER_ADMIN, MINISTRY_ADMIN, AGENCY_MANAGER, MANAGER, AUDITOR.
5. **Single database, multiple schemas** — django-tenants routes requests to the correct schema based on domain or user profile.

## Quick Reference — Files to Edit for Common Changes
| Change | File(s) |
|--------|---------|
| Add a new app | `config/settings.py` (add to SHARED_APPS or TENANT_APPS) |
| Change JWT expiry | `config/settings.py` (SIMPLE_JWT dict) |
| Change rate limits | `config/settings.py` (REST_FRAMEWORK dict) |
| Add a permission/role | `authentication/models.py` (ROLE_CHOICES) + `authentication/api_permissions.py` |
| Add a new API endpoint | `*/api_views.py` + `*/api_urls.py` + `*/api_serializers.py` |
| Change schema routing | `config/settings.py` (SHOW_PUBLIC_IF_NO_TENANT_FOUND) + `tenants/models.py` (Domain) |
| Deploy to production | See Part 2 Chapter 15 for full checklist |
| Connect external system | See Part 1 Chapter 7 for full guide |
